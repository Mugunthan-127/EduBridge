EDUBRIDGE
UI Element and Behavior Specification
Documents every element present in EduBridge_Prototype.jsx, section by section, and precisely how each one performs — what it shows, what triggers it, and what state changes as a result.

1. Global App Shell
Present on every screen regardless of which role is active. This is the persistent frame the five role views sit inside.
1.1 Status strip (top bar)
A dark, monospace-font strip fixed above all content. Its job is to make the offline-first architecture visible at all times, not just documented in text.
Element
Type
Behavior — exactly how it performs
Connectivity toggle
Button
Shows a Wi-Fi icon and "online"/"offline" label. Clicking flips a single online/offline flag for the whole app. Every other component (video pill, AI tutor tier, sync behavior) reads this same flag, so one click demonstrates the full offline-first behavior consistently across screens.
Sync queue counter
Live counter + icon
Reads the length of an in-memory queue array that every write action (quiz submit, material upload, mesh advertise, review decision, package push) appends to. Shows a spinning icon while a sync is in progress. While offline and the queue is non-empty, appends "(retry in Ns)" using the current backoff value.
Mesh peers counter
Live counter + icon
Displays the current simulated peer count. Starts at 2; increases by 3 whenever the teacher's "Advertise to mesh" action runs, modeling nearby devices picking up the advertised content.
Last sync counter
Live counter
Ticks upward once per second showing seconds since the last successful sync, recalculated from a stored timestamp. Resets to 0 the moment a sync completes successfully.
Sync now button
Button
Manually triggers the same flush logic that runs automatically when connectivity is restored. If online: after an 800ms simulated network delay, clears the queue, resets the backoff counter to 2s, and resets the last-sync timer. If offline: the queue is left untouched and the backoff counter doubles (2 to 4 to 8 to 16, capped at 32 seconds), modeling exponential backoff on repeated failures.
1.2 Role switcher
A row of five pill buttons directly under the header.
Element
Type
Behavior — exactly how it performs
Role pills
Button group
Exactly one of Student / Teacher / Parent / School admin / District is active at a time. The active pill is filled dark; inactive pills are outlined. Clicking a pill swaps the entire content area below to that role's view — each role's internal state (mastery, queue items already pushed, etc.) persists in memory even after switching away and back, since it is held in the top-level component, not remounted.

2. Student Dashboard
Two-column layout: a wider left column for learning activity, a narrower right column for progress and the AI tutor.

Element

Per-topic progress bar
Bar + % label
One bar per topic (Fractions, Reflection of light, Linear equations). Width is literally the student's current BKT mastery probability for that topic, redrawn with a 0.5s CSS transition whenever it changes — so a correct quiz answer visibly grows the bar in real time.

7. Implementation Notes and Honest Limitations
• All state in this prototype lives in React component state (useState/useEffect) and resets on page reload. Claude.ai artifacts cannot use localStorage, sessionStorage, or IndexedDB, so this build intentionally simulates the offline storage layer rather than persisting it — the real Dexie/IndexedDB schema is implemented separately in the core code package (EduBridge_Core.zip).
• The BKT mastery formula used in the Student dashboard is the identical algorithm to frontend/src/mastery/bktEngine.js in the core package, so behavior demonstrated here (mastery rising with correct answers, recommending the weakest topic) is not just illustrative — it is the same math, verified separately by that package's automated tests.
• The AI tutor's four tiers are simulated with canned response text rather than live Gemini/local-model calls, so the connectivity-driven tier switching can be demonstrated without an API key; wiring real calls means replacing the two response-generation branches in the AITutor component.
• The Local Content Mesh's "Advertise to mesh" action is a state-change simulation (peer count increment), not a real WebRTC/BroadcastChannel transfer; the actual transfer protocol, including checksum verification, is implemented and tested in frontend/src/mesh/contentMesh.js in the core package.
• Multi-tenant scoping (a School Admin only seeing their own school, a District Admin seeing all schools in their district) is not enforced in this prototype — every role's data is shown unconditionally for demo purposes. Real tenant scoping is enforced server-side in the backend skeleton's data model.

Priority key: Must = required for a working demo · Should = required for a complete system · Could = enhancement if time permits
